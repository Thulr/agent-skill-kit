# Research Report — Writing minimal, concise, legible code (and why coding agents fail at it)

**Date:** 2026-06-17
**Depth mode:** deep-dive (5-worker fan-out, ~40 sources, primary-source-first with citation chasing)
**Methodology grounding:** `research` skill, report frame (see that skill's `skill.json.inspired_by`)
**Purpose:** Source base for a new `informed-skills` catalog skill on writing minimal/concise/legible code and counteracting agent "slop."

---

## 1. Research question

What does the literature — both the craft canon and recent empirical studies — say about writing **minimal, concise, yet legible** code, and **why do LLM coding agents systematically fail to do so** (bloat, duplication, failure to reuse, over-engineering, poor architectural choices)?

- **In scope:** the simplicity/minimalism philosophy in software; named principles (YAGNI, DRY/WET, the wrong abstraction, KISS, accidental vs essential complexity, deep modules, Gall's Law); empirical evidence on agent/LLM code pathologies (duplication, churn, reuse failure, technical debt, package hallucination); evidence-backed counter-techniques; the conciseness↔legibility line (minimal ≠ terse/clever/golfed).
- **Out of scope:** general prompt-engineering tutorials; non-code writing craft except as a cited analog (Strunk & White); vendor marketing treated as load-bearing evidence; language-specific style guides.
- **Audience / use:** the maintainer of this catalog, to ground a new skill. Every load-bearing claim here is cited and confidence-tagged so it can be reused verbatim in a `SKILL.md` / `skill.json` `inspired_by` list. This repo's invariant is that skills are *grounded in cited literature, never uncited* — so attribution accuracy is itself a deliverable, and dubious attributions are flagged, not laundered.

## 3. Background

Terms a reader needs before §4–§6:

- **Simple vs easy** *(Hickey 2011 — H)*: *simple* comes from *sim-plex*, "one fold/braid"; its opposite is *complex*, "braided together." *Easy* derives from a root meaning "to lie near, nearby." So **simple is about lack of interleaving (an objective property of the artifact); easy is about familiarity/proximity (relative to a person).** "Complecting" = interleaving/entwining concerns that needn't be joined.
- **Complexity** *(Ousterhout 2018 — H)*: "anything related to the structure of a software system that makes it hard to understand and modify the system." Note this is a *cognitive* definition — complexity is whatever raises the cost of understanding and change.
- **Accidental vs essential complexity** *(Brooks 1986/87 — H)*: *essence* = "the difficulties inherent in the nature of the software"; *accidents* = "those difficulties that today attend its production but that are not inherent." Minimalism attacks the accidental; it cannot remove the essential.
- **DRY is about knowledge, not text** *(Hunt & Thomas 1999 — H)*: "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system." The authors gloss it as "the duplication of *knowledge, of intent*" — two identical-looking fragments encoding *different* decisions are not a DRY violation.
- **Code churn**: code that is revised or reverted shortly after being committed (GitClear operationalizes it as edited/reverted within ~2 weeks). High churn is a rework/waste proxy.
- **"Slop"**: informal term for low-quality, bloated, or redundant generated output; in dependency-land it produced **"slopsquatting"** (hallucinated package names that attackers then register).

Grouped by theme. Each load-bearing claim carries a citation at the point of claim and an H/M/L confidence tag (confidence = strength of evidence, not significance).

### 4A. The craft canon: simplicity is hard, objective-ish, and the core job

- **There are two ways to build a design; the simple one is harder and the only one that makes defects visible.** "I conclude that there are two ways of constructing a software design: One way is to make it so simple that there are obviously no deficiencies and the other way is to make it so complicated that there are no obvious deficiencies. The first method is far more difficult." *(Hoare, 1980 ACM Turing Award lecture "The Emperor's Old Clothes," CACM 24(2), 1981 — H)*
- **Dealing with complexity is the most important challenge in software design** *(Ousterhout 2018 — H)*; it is "incremental: you have to sweat the small stuff." Complexity accrues line by line, so minimalism is a per-change discipline, not a one-time cleanup.
- **Reliability requires simplicity, and the programmer's skull is small.** "The competent programmer is fully aware of the strictly limited size of his own skull; therefore he approaches the programming task in full humility, and … avoids clever tricks like the plague." *(Dijkstra, EWD340 "The Humble Programmer," 1972, CACM 15(10) — H)* The aphorism "Simplicity is prerequisite for reliability" is attributed to EWD498 (1975) but appears there as a handwritten margin annotation, not body text *(M — cite with that caveat)*.
- **Minimal interface ("deep modules"), not minimal line count, is the target.** Ousterhout: "it is more important for a module to have a simple interface than a simple implementation"; deep modules pack "lots of functionality, simple interface." He coins **"classitis"** for the reflex that classes/functions must be tiny — which spawns many *shallow* units whose accumulated interfaces *raise* total system complexity *(Ousterhout 2018 — H for the deep-module/interface quotes; M for the exact one-sentence "classitis" definition)*. **This is the single most important nuance: "minimal" means minimal surface/complexity, not fewest lines or smallest units.**
- **Write below your maximum cleverness so you can still debug it.** "Everyone knows that debugging is twice as hard as writing a program in the first place. So if you're as clever as you can be when you write it, how will you ever debug it?" *(Kernighan & Plauger, The Elements of Programming Style, 2nd ed. 1978, ch. 2 — H)*. ⚠️ The snappier "…you are, by definition, not smart enough to debug it" form is a **popular paraphrase**, not the book's words — use the original or label it a paraphrase *(M/L on the paraphrase being verbatim)*.
- **Do one thing well; don't write the big program unless proven necessary.** Unix philosophy: "Write programs that do one thing and do it well." *(McIlroy; the exact wording is the Salus/Raymond 1994/2003 phrasing — McIlroy's 1978 original is "Make each program do one thing well" — M on which phrasing/venue)*. Raymond's Rule of Parsimony: "Write a big program only when it is clear by demonstration that nothing else will do," and Rule of Clarity: "Clarity is better than cleverness." *(E.S. Raymond, The Art of Unix Programming, 2003 — H for the rules)*
- **Bloat is a choice, and lean software is possible.** "Reducing complexity and size must be the goal in every step — in system specification, design, and in detailed programming." *(Wirth, "A Plea for Lean Software," IEEE Computer 28(2), 1995 — M; venue strong, quotes confirmed via one good secondary, primary PDF didn't render)*
- **"Worse is Better" ranks implementation simplicity above interface simplicity, correctness, and completeness** — "Simplicity is the most important consideration in a design." *(Gabriel, "The Rise of Worse is Better," 1991 — H for the text)*. But Gabriel was ambivalent and later argued the opposite under a pseudonym, so it is a *described phenomenon*, not a clean prescription *(see §5)*.
- **The non-software canon for "minimal = nothing left to remove":** "Perfection is attained not when there is nothing more to add, but when there is nothing more to remove" *(Saint-Exupéry, Terre des Hommes / Wind, Sand and Stars, 1939 — H on work + French original "rien à retrancher"; M on which English translation, several circulate)*.

### 4B. Named minimalism principles — and the caveat each carries

The principles matter *with their caveats*, because the naive form of each is itself a cause of the over-engineering this skill should prevent.

- **YAGNI** ("You Aren't Gonna Need It"): "Always implement things when you actually need them, never when you just foresee that you need them" *(XP / c2 wiki, attrib. Ron Jeffries c.1999 — H on wording, M on single-author attribution)*. **Caveat (load-bearing):** Fowler scopes it precisely — "Yagni only applies to capabilities built into the software to support a presumptive feature, it does not apply to effort to make the software easier to modify… Yagni requires (and enables) malleable code." *(Fowler, "Yagni," 2015 — H)*. So YAGNI cuts speculative *features*, not code health/refactoring.
- **DRY** = single authoritative representation of each *piece of knowledge* *(Hunt & Thomas 1999 — H)*. **Caveat:** it is about knowledge/intent, not textual duplication; mechanically de-duplicating look-alike code is the #1 road to the wrong abstraction.
- **The wrong abstraction** is costlier than duplication: "duplication is far cheaper than the wrong abstraction"; "prefer duplication over the wrong abstraction." *(Metz, "The Wrong Abstraction," 2016, from her RailsConf 2014 talk "All the Little Things" — H)*. The remedy when stuck with one: "the fastest way forward is back" — re-inline, then re-abstract from clearer requirements. Duplication's cost is linear and obvious; a wrong abstraction's cost compounds and is defended by sunk cost.
- **Rule of Three:** "The first time… you just do it. The second time… you wince… but you do the duplicate thing anyway. The third time… you refactor." → "Three strikes and you refactor." *(Fowler, Refactoring, 1999, crediting Don Roberts — H on attribution/gist; M on exact book-page wording)*. It exists *because* abstracting on the second occurrence risks the wrong abstraction — "three" means "wait for enough evidence that it's the same knowledge," not a literal counter.
- **Accidental vs essential complexity** *(Brooks, "No Silver Bullet," 1986/87 — H; quotes via faithful reproduction, primary PDF 403-blocked)*: "the hard part of building software [is] the specification, design, and testing of this conceptual construct, not the labor of representing it." Minimalism removes accidental complexity; it cannot dissolve essential complexity — don't oversell simplicity as a fix for inherently hard problems.
- **Gall's Law:** "A complex system that works is invariably found to have evolved from a simple system that worked. A complex system designed from scratch never works… You have to start over with a working simple system." *(Gall, Systemantics, 1975 — H on wording; M on primary-page fetch)*. Argues for growing complexity from a working simple core, not big-design-up-front.
- **"The best code is no code at all… Every new line of code you willingly bring into the world is code that has to be debugged, code that has to be read and understood, code that has to be supported."** *(Atwood, Coding Horror, 2007 — H)*. The rigorous, citable version of "LOC is a cost, not an output": "if we wish to count lines of code, we should not regard them as 'lines produced' but as 'lines spent'." *(Dijkstra, EWD1036, 1988 — H)*.
- **KISS** ("Keep It Simple, Stupid"), attributed to Kelly Johnson at Lockheed Skunk Works — **triage as folklore** *(L on the Johnson attribution; the acronym is documented to the US Navy 1960 and "keep it short and simple" to 1938)*. Use as a sticky label, not an authority.
- **SRP / single responsibility** = "one reason to change" (cohesion of change), not "does one literal thing" *(R.C. Martin, popularized 2003 — H)*. Over-applied, it fragments code into too many tiny classes — itself a minimalism smell.

### 4C. Empirical evidence: agents under-reuse, over-duplicate, and accrue debt

This is the backbone of the *problem statement*. Triage legend: **PR** = peer-reviewed/arXiv; **INST** = institutional; **VENDOR** = commercial vendor (corroborating only). The strongest claims rest on before/after-commit *delta* studies, not vendor correlations.

- **AI pull requests are ~1.87× more redundant and skip reuse.** Average Max Redundancy 0.2867 (AI) vs 0.1532 (human), **p<0.001** (semantic/Type-4 redundancy). Verbatim conclusion: "LLM Agents frequently disregard code reuse opportunities, resulting in higher levels of redundancy" and "tend to generate redundant code instead of reusing existing code." Reviewers nonetheless rated AI PRs *more* favorably — a blind-spot signal. *(Huang et al., "More Code, Less Reuse," MSR 2026; 3,858 Python PRs + 617 from crewAI, AIDev dataset — H. Strongest single source for the core thesis.)*
- **Agents almost never refactor to deduplicate or reuse.** Among refactoring motivations, agents pursue **duplication elimination 1.1% vs humans' 13.7%**, and **reuse/repurpose 4.6% vs 12.9%** — the two DRY/reuse-improving motivations are the *least* common for agents and *prominent* for humans. *(Horikawa et al., "Agentic Refactoring," arXiv 2025; 154,511 refactoring instances across 1,613 repos; Codex-dominated, Java — H, with single-language caveat.)*
- **AI commits routinely introduce code smells/debt that persists.** Of 484,606 issues introduced by AI commits, **89.1% were code smells**; **>15% of every AI assistant's commits introduce ≥1 issue (17.3% Copilot → 28.7% Gemini)**; **24.2% of AI-introduced issues still survive at HEAD**. Static analysis run immediately before/after each AI commit (true delta design). *(Liu et al., "Debt Behind the AI Boom," arXiv 2026; 304,362 verified AI-authored commits, 6,275 repos — H.)*
- **~1 in 5 LLM-recommended packages don't exist.** Hallucinated-package rate averages **5.2% (commercial models)** to **21.7% (open-source models)**; 205,474 unique hallucinated names observed. *(Spracklen et al., "We Have a Package for You!," USENIX Security 2025; 576K samples, 16 LLMs — H.)* This seeded **"slopsquatting"** *(coined Seth Larson, 2025; earlier Lasso Security/Lanyado 2024 found a hallucinated "huggingface-cli" drew >30K downloads — VENDOR, corroborating)*.
- **The mechanism behind failure-to-reuse: "lost in the middle."** LLM accuracy is highest when relevant info sits at the start/end of context and "degrades significantly when [it] is in the middle"; performance drops as context grows, even for long-context models. *(Liu et al., "Lost in the Middle," TACL 2024 — H.)* This explains *why* an agent with the repo in context still re-implements an existing function — but it is the "why," not a direct measurement of code-reuse failure (which §4C-items-1/2 supply).
- **Volume up, discipline down (institutional).** A 25% increase in AI adoption is associated with **−7.2% delivery stability** and **−1.5% throughput**, even as individual-level documentation/code-quality self-ratings rise; DORA's own framing is that AI "makes it easier to write more code," raising batch size and instability. **39% of respondents report little-to-no trust in AI-generated code.** *(Google DORA, State of DevOps 2024 — M/H; association not causation, self-reported adoption.)*
- **Authors don't notice the cost.** Developers with an AI assistant "wrote significantly less secure code" on 4 of 5 tasks *and* were "more likely to believe their code was secure." *(Perry et al., "Do Users Write More Insecure Code with AI Assistants?," ACM CCS 2023; N=47 — M/H; small N, older model.)* Pairs with the reviewer-overrating finding above: slop is invisible to the people best placed to catch it.
- **Verbosity/length instability (weaker).** Verbosity bias (preferring longer output even when brevity is asked) is documented; code correctness differed on ~12% and *efficiency* on ~39% of tasks based purely on prompt wording. *(Jiang et al., "Linguistic Bias in LLM-based Code Generation," arXiv 2024 — M.)* No single clean benchmark proving "LLMs add unnecessary abstractions" was found — over-engineering is *inferred* from smell density + verbosity, not directly measured (see §8).
- **VENDOR corroboration (not load-bearing):** GitClear's 2025 report (211M changed lines) finds churn rising (3.1%→5.7%, 2020–2024), "moved"/refactored lines falling (24.1%→9.5%), copy-pasted code (8.3%→12.3%) overtaking moved code in 2024, and a ~4–8× rise in cloned blocks. *(GitClear 2025 — M; vendor, temporal correlation only, AI-attribution method undisclosed — cannot be sole support for a causal claim. Use it for scale/color; anchor causal claims on the peer-reviewed deltas above.)*

### 4D. Counter-techniques (evidence-backed)

- **Reuse-first / search before you write.** "Software reuse is the process of creating software systems from existing software rather than building software systems from scratch." *(Krueger, "Software Reuse," ACM Computing Surveys 24(2), 1992 — H)*. Concrete agent move: grep the codebase + dependency surface for an existing helper before writing a new one (the "grep first" phrasing is folklore; Krueger is the academic anchor).
- **We systematically overlook subtraction — the highest-value finding for "why agents add."** Across 8 experiments, people "were less likely to identify advantageous subtractive changes when the task did not cue them to consider subtraction"; "additive ideas come to mind quickly and easily, but subtractive ideas require more cognitive effort." *(Adams, Converse, Hales & Klotz, "People systematically overlook subtractive changes," Nature 592, 2021 — H.)* Implication: subtraction must be *explicitly prompted* ("what can be removed?") or it won't surface — for humans or agents.
- **Make it work, then make it right — in separate steps.** Refactoring = "changing a software system in such a way that it does not alter the external behavior of the code yet improves its internal structure" *(Fowler, Refactoring, 1999 — H)*. Beck's two rules of TDD: "1. Write new code only if an automated test has failed. 2. Eliminate duplication"; the Refactor step explicitly "eliminate[s] all of the duplication created in merely getting the test to work." *(Beck, TDD by Example, 2002 — H.)* Concrete: get green, then dedupe/tidy without changing behavior — never in the same diff.
- **Complexity metrics as smell detectors.** Cyclomatic complexity V(G)=E−N+2P measures independent paths and *testability* *(McCabe, IEEE TSE, 1976 — H)*; Cognitive Complexity measures *understandability* (increment on breaks in linear flow, extra for nesting, ignore shorthand that readably condenses) because cyclomatic "works very well for measuring testability, but not for maintainability" *(Campbell, SonarSource / ACM TechDebt 2018 — H)*. Treat rising numbers as a refactor signal — but they are smell detectors, not goals (§5: Goodhart).
- **Review for "not more complex than it needs to be."** Reviewers should ask "Is the CL more complex than it should be?" at "individual lines, functions, and classes," and flag over-engineering — code "more generic than it needs to be, or … functionality that isn't presently needed." *(Google Engineering Practices, "What to look for in a code review" — H.)* The standard: approve once it "definitely improves the overall code health," not when perfect.
- **Naming is the legibility lever.** Names should be "precise, unambiguous, and intuitive"; difficulty naming something signals it "represent[s] too many things" and should be split *(Ousterhout 2018 — H on principle, M on verbatim)*. The famous "two hard things… naming things" line is **folklore attributed to Phil Karlton with no primary source** *(Fowler documents the missing provenance — H that it's unverifiable)*.
- **Brevity comes from a second pass, not the first draft.** "I have made this longer than usual because I have not had time to make it shorter" — genuinely **Pascal, Lettres Provinciales, Letter 16, 1657**, and **misattributed to Mark Twain** *(Quote Investigator — H)*. "Plan to throw one away; you will, anyhow" *(Brooks, Mythical Man-Month, 1975 — H; note Brooks softened this toward iterative delivery in 1995)*.
- **Chesterton's Fence — the guardrail on deletion.** Don't remove what you don't understand: the reformer who says "I don't see the use of this; let us clear it away" should be told "Go away and think… when you can come back and tell me that you do see the use of it, I may allow you to destroy it." *(Chesterton, The Thing, 1929 — H; the snappy "never take a fence down until you know why it went up" is a later paraphrase.)*
- **Leave it cleaner; tidy separately.** Boy Scout Rule: "Always leave the campground cleaner than you found it" → "check a module in cleaner than when you checked it out" *(R.C. Martin, 97 Things…, 2010 — H)*. "Tidyings" = "small, structural code improvements that should be separated from behavioral changes" *(Beck, Tidy First?, 2023 — H on framing, M on verbatim)*.

### 4E. Conciseness ≠ terseness: the legibility line

- **Code's primary audience is human readers.** "programs must be written for people to read, and only incidentally for machines to execute." *(Abelson & Sussman, SICP, Preface to the 1st ed., 1985 — H.)* And Knuth: "Instead of imagining that our main task is to instruct a *computer* what to do, let us concentrate rather on explaining to *human beings* what we want a computer to do." *(Knuth, "Literate Programming," The Computer Journal 27(2), 1984, p.97 — H.)*
- **You read code far more than you write it — so optimize for reading.** "the ratio of time spent reading versus writing is well over 10 to 1." *(R.C. Martin, Clean Code, 2008, ch.1 — H.)*
- **Concise means "no wasted elements," not "fewest characters."** Strunk's Rule 17: "Vigorous writing is concise. A sentence should contain no unnecessary words… for the same reason that a drawing should have no unnecessary lines and a machine no unnecessary parts. This requires not that the writer make all his sentences short… but that every word tell." *(Strunk & White, The Elements of Style — H.)* The programming-style tradition consciously borrows this: *The Elements of Programming Style* "pays explicit homage, in title and tone, to The Elements of Style" *(— H)*.
- **Don't be too clever (the anti-golf maxims).** "Write clearly — don't be too clever" and "Don't sacrifice clarity for small gains in efficiency." *(Kernighan & Plauger, The Elements of Programming Style — H.)*
- **Even the strongest terseness advocate draws the line at characters.** "succinctness is power… except in pathological examples you can treat them as identical," BUT "What I'm looking for are programs that are very dense according to the metric of 'elements'… not merely programs that are short because delimiters can be omitted and everything has a one-character name." *(Graham, "Succinctness is Power," 2002 — H; editorial.)* So *succinctness measured in semantic elements* is the goal; *terseness measured in characters* (code golf) is not.
- **Readability is measurable and predicts defects.** A model from "a simple set of local code features" was "80% effective… at predicting readability judgments" and "correlates strongly with three measures of software quality: code changes, automated defect reports, and defect log messages." *(Buse & Weimer, "Learning a Metric for Code Readability," IEEE TSE 36(4), 2010 — H.)* Follow-up work attributes the strongest negative weights to average line length, identifier count, and punctuation density — i.e., **over-dense lines measurably reduce readability** *(Scalabrino et al. 2018, citing Buse & Weimer — M on exact phrasing)*.
- **Elegance is functional, not decorative.** "in computing, elegance is not a dispensable luxury but a quality that decides between success and failure." *(Dijkstra, EWD1284, 1999 — H.)* ⚠️ This is **EWD1284, not EWD1300** (a common miscitation); EWD1300 separately frames brevity as achievable "without committing the sin of omission" — itself a clean statement of the concise-but-not-cryptic line.

## 5. Key debates and open questions

- **DRY vs the wrong abstraction (the central craft tension).** DRY says eliminate duplicated knowledge; Metz says a wrong abstraction is costlier than duplication. *Resolution in the literature:* DRY is about *knowledge, not text*, gated by the Rule of Three. A minimalism skill must hold both — **minimal ≠ maximally de-duplicated.** (Interpretive disagreement, resolvable.)
- **YAGNI vs extensibility/OCP.** YAGNI says don't build speculative generality; "design for change"/Open-Closed pushes toward extension points. Fowler resolves it: YAGNI applies to *presumptive features*, not to keeping code malleable. **Directly relevant to this repo:** the global `CLAUDE.md` prefers "plugin/extension point patterns… New behavior should be addable without modifying existing code (OCP)." That is a real, citeable tension the new skill must reconcile honestly, not paper over — extension points that serve a *present* need are code health; extension points for an *imagined* future are the speculative generality YAGNI warns against.
- **"Worse is Better" — prescription or lament?** Genuinely contested; Gabriel himself was ambivalent and later argued the opposite under a pseudonym. Cite it as a *described phenomenon* (simple-but-crude designs spread), not a clean endorsement of "always pick simple."
- **Interface vs implementation simplicity.** Ousterhout and Gabriel's "right thing" school prioritize *interface* simplicity (deep modules); Gabriel's "worse is better" prioritizes *implementation* simplicity. "Minimal code" must name which — minimal *interface* (deep module) ≠ minimal *implementation* (fewest lines).
- **Is "minimal" = "small units"?** No — Ousterhout's "classitis"/deep-module argument is a direct rebuttal: too many tiny shallow functions/classes *raise* aggregate complexity. The skill should distinguish minimal-surface from minimal-line-count, or it will produce the very fragmentation it's trying to prevent.
- **Metric gaming (Goodhart).** Optimizing cyclomatic/cognitive complexity per-function can just *move* complexity (extract a 30-line method into ten 3-line methods → lower per-function scores, more indirection to trace). Metrics are smell detectors, not targets.
- **Delete vs Chesterton's Fence.** Subtraction is under-explored (Adams et al.) *and* reckless deletion is dangerous (Chesterton). Resolution: deliberately *consider* subtraction (because it's cognitively skipped) but *gate it on understanding* (because the fence may be load-bearing). Pair them; never cite one alone.
- **Productivity vs quality (contested empirics).** The METR RCT found 16 experienced OSS devs were **19% slower with AI while believing they were 20% faster** *(Becker et al., arXiv 2025)* — yet field experiments at Microsoft/Accenture report throughput *gains*. **Open question / honest framing:** speedup is real in some settings; the *maintainability* cost (churn, duplication, debt) is the more consistent finding, and it's largely invisible to the author. Don't overclaim "AI makes you slower."
- **Correlation vs causation.** GitClear and DORA are associations over an adoption window; the before/after-commit deltas (More Code Less Reuse; Agentic Refactoring; Debt Behind the AI Boom) are methodologically stronger. Anchor category claims on the deltas.

## 6. Implications — for crafting the skill

Marked **[sourced]** (a cited source states it) or **[inferred]** (synthesis across the sources named).

- **[inferred from §4A+§4E]** *Thesis the skill should encode:* **minimal = "nothing left to take away" + deep modules + reuse-first + legible-for-humans — NOT fewest lines, smallest units, or cleverest one-liner.** This single reframing (Saint-Exupéry/Strunk "no waste" + Ousterhout "deep modules/complexity" + Graham "elements not characters" + SICP/Knuth "for humans") prevents the skill from being misread as code-golf or as "make everything tiny."
- **[sourced]** *The problem is now empirically grounded, not folklore:* agents measurably under-reuse and over-duplicate *(More Code Less Reuse: 1.87×, p<0.001; Agentic Refactoring: 1.1% vs 13.7% dedup)*, accrue persistent smells/debt *(Debt Behind the AI Boom)*, and hallucinate dependencies *(USENIX 2025)* — and the author/reviewer doesn't notice *(Stanford; reviewers over-rate AI PRs)*. The mechanism is partly retrieval *(Lost in the Middle)*. This is strong `inspired_by` material and a credible activation rationale.
- **[inferred from §4C+§4D]** *Candidate operating procedure (the actionable core):*
  1. **Search before you write** — grep the repo + deps for an existing implementation/abstraction before adding one *(Krueger; counter to Lost-in-the-Middle reuse failure)*.
  2. **Ask "what can be removed?" explicitly** — subtraction won't surface on its own *(Adams et al.)*.
  3. **Don't abstract before the third occurrence**; prefer duplication to a guessed abstraction *(Metz; Rule of Three)*.
  4. **Scope YAGNI to features, not malleability** — cut speculative features, keep code refactorable *(Fowler)*.
  5. **Make it work, then tidy in a separate diff** — never bundle behavior + structure changes *(Beck; Fowler)*.
  6. **Gate deletion on understanding** — git-blame/tests/issue before removing "dead-looking" code *(Chesterton)*.
  7. **Treat rising cyclomatic/cognitive complexity as a refactor signal, not a target** *(McCabe; Campbell)*.
  8. **Name precisely; a name that needs a comment or reaches for `data`/`tmp`/`manager` is a design smell** *(Ousterhout)*.
  9. **Budget a second pass to shorten** — the first working version is the throwaway pilot *(Pascal; Brooks)*.
- **[inferred]** *Tensions the skill must hold explicitly* (don't pick one side silently): DRY↔wrong-abstraction; YAGNI↔OCP/extension-points (note this repo's own OCP preference); delete↔Chesterton; minimal-lines↔deep-modules; succinct-elements↔terse-characters; metric-as-signal↔metric-as-target.

# PART II — The second force: architecture for parallel multi-agent execution, and the reconciliation

**Added:** 2026-06-17 (follow-up pass)
**Trigger:** the maintainer reframed the problem as *two* forces — (1) keeping agents minimal/anti-slop [Part I], and (2) structuring projects so **many agents can execute in parallel**, which *requires* architectural investment. The tension between "don't over-build" (force 1) and "invest in modular boundaries" (force 2) is the thing the skill must reconcile.

**Method note (provenance):** this pass mined a **private** AI knowledge wiki to find the right *public* sources and framing. Per a standing constraint, **the private wiki is never cited** in the report or in any published skill — it is a shaping input only. Every citation below is a public source the wiki pointed to, re-checked for existence by the research workers; unverifiable pointers are quarantined in §II.3.

## II.1 Current state — what makes a codebase safe for N parallel agents

- **Conway's Law: the system mirrors the org that builds it.** "Any organization that designs a system… will produce a design whose structure is a copy of the organization's communication structure." *(Conway, "How Do Committees Invent?," Datamation, 1968 — H, primary verified.)* A fleet of loosely-coupled agents *is* a loosely-coupled org → expect loosely-coupled output, which is fine **only if** the codebase already enforces clean module boundaries.
- **The Mirroring Hypothesis (the measured form).** Across matched pairs, "the product developed by the loosely-coupled organization was significantly more modular than the product from the tightly-coupled organization." *(MacCormack, Rusnak & Baldwin, "Exploring the Duality between Product and Organizational Architectures," HBS WP 2008 / Research Policy 41(8), 2012 — H, verified.)* The strongest empirical anchor that **parallel agents require modularity** — not as taste, but as a product-org duality.
- **Design rules vs hidden modules (the parallelization mechanism).** A modular design = (1) stable **design rules** (interfaces/standards everyone obeys), (2) **hidden modules** that obey them and can be built/replaced independently with no ongoing consultation, (3) a **systems-integration-and-testing** step. *(Baldwin & Clark, Design Rules: The Power of Modularity, MIT Press, 2000; + "Modularity in the Design of Complex Engineering Systems," 2006 — H, verified.)* This is the cleanest vocabulary for agent fan-out: **one owner freezes the contracts; each agent owns a hidden module; CI reconciles.**
- **Modularity is necessary but NOT sufficient.** Parallel projects "fall apart particularly during integration" when interface specs "lack essential detail." *(Herbsleb & Grinter, "Architectures, Coordination, and Distance: Conway's Law and Beyond," IEEE Software 16(5), 1999 — H, verified.)* The critical counterweight: if the interface isn't explicit and **frozen**, N agents don't reduce work — they relocate the collision to merge/integration, which is costlier. This is the empirical root of "mergeable by default."
- **Same design-rule layer ⇒ independently parallelizable.** "Software modules located within the same layer of the hierarchy suggest independent, hence parallelizable, tasks"; validated on Apache Ant — cross-layer pairs communicate significantly more than within-layer pairs. *(Wong, Cai et al., "Design Rule Hierarchies and Parallelism in Software Development Tasks," ASE 2009 — H, verified.)* Gives the skill a **concrete work-assignment rule**: derive a layer/dependency map, hand same-layer modules to different agents, serialize across layers.
- **Information hiding bounds a reader to one module — human or agent.** Information hiding lets you "study the system one module at a time." *(Parnas, "On the Criteria To Be Used in Decomposing Systems into Modules," CACM 15(12), 1972 — H, seminal.)* **The hinge for the whole reconciliation:** a boundary that bounds *human* attention is the same boundary that fits an *agent's* context window. One investment serves both readers.
- **The agent-native restatement (2025).** "The opportunities offered by LLM coders demand a reevaluation of how software is structured"; decompose into independent **concepts** + explicit **synchronizations** so a change touches one visible unit. *(Meng & Jackson, "What You See Is What It Does: A Structural Pattern for Legible Software," Onward! 2025 / arXiv 2508.14511 — H, verified.)* ⚠️ "legibility-first design" is the wiki's label, **not** the paper's title — cite the real title.
- **Handing agents an explicit structure map is a large, measured win.** Providing a deterministic repository/dependency graph "improves mean accuracy by 12.2% and reduces completion time by 53.9%" across Claude Code, Cursor, and Codex on 8 repos. *(Cherny-Shahar & Yehudai, "Repository Intelligence Graph," arXiv 2601.10112, 2026 — H, verified.)* The same graph that helps one agent is how you **partition** work and detect when two agents' modules touch.
- **Compute blast radius before dispatching an edit.** Repository-level editing needs explicit "incremental dependency analysis, change may-impact analysis, and adaptive planning"; CodePlan passes 5/7 repos where baselines pass 0. *(Bairi et al., "CodePlan: Repository-level Coding using LLMs and Planning," FSE 2024 / Microsoft Research — H, verified.)* The "blast radius / may-impact" citation behind non-colliding parallel edits.
- **Interface quality (not just existence) gates throughput.** Interface design anomalies (duplication, unused methods, ambiguity) "are reliable symptoms of poor interface design" and measurably harm maintainability across 9 OSS projects. *(Abdeen et al., arXiv 1309.7950, 2013 — H, verified; secondary support.)*
- **Practitioner confirmation — "the architectural reason is concurrency."** Anthropic's long-running-agent engineering: multi-agent patterns work best when the **hand-off is a concrete, testable contract** — generator and evaluator negotiate "done" as files on disk, graded against an acceptance spec; orchestrator-worker subagents run "in parallel" (reported 90.2% improvement over single-agent) **but need tight scoping or they over-spawn and chase nonexistent sources.** *(Anthropic Engineering: "Building Effective Agents," 2024; "How we built our multi-agent research system," 2025; "Harness design for long-running application development," 2026 — H, verified.)* This is the honest cost note that ties back to minimalism: parallelism is not free.

## II.2 The reconciliation (the skill's core thesis)

The two forces are **the same coin**. Four lines, citeable end-to-end:

1. **[inferred from Baldwin & Clark + Fowler]** *Invest in interfaces, not implementations.* Freeze a thin set of **design rules** (stable contracts) once; everything behind them is a **hidden module** an agent can build, replace, or throw away cheaply. You *over-invest in the boundary, under-invest in the code volume.* This is exactly Fowler's YAGNI carve-out [Part I §5]: the boundary serves a **present** need (parallel execution + legibility = "malleability"), so it is code health, not speculative generality. Over-built extension points for *imagined* features are still slop.
2. **[sourced — Herbsleb & Grinter]** *Boundaries are what let you stay minimal under parallelism.* Without explicit, stable interfaces, N agents relocate the collision to integration. So the boundary investment *prevents* the big over-build (the merge-conflict/integration cleanup) — it isn't an addition to it.
3. **[sourced — Parnas → Meng & Jackson]** *One legibility investment serves two readers.* The module boundary that bounds human attention is the boundary that fits an agent's context window. You don't bolt agent-specific architecture onto human architecture — you make the existing boundary **legible and mechanically enforced.** (This is why "just adopt Clean/Hexagonal architecture for agents" is the wrong move: don't add ceremony; make the substrate visible and gate it.)
4. **[inferred — Gomes/Cursor + Nisi + Anthropic "assumptions expire"]** *Express the harness in the cheapest medium that still enforces — and delete scaffolding as models improve.* Cursor replaced ~12–15K LoC of native worktree machinery with ~200 lines of markdown *(David Gomes, "Replacing 12K LoC with a 200-LoC Skill," AI Engineer 2026 — talk verified; line counts are stage estimates)*; Nisi cut 10,000+ lines of auto-generated skills to 553 lines of "gotchas," dropping an eval suite from 68→6 minutes with better pass rates *(Nick Nisi, "How I deleted 95% of my agent skills…," AI Engineer 2026 — talk verified; one team's evals, not independent)*. **But bounded by enforcement:** Gomes's markdown lost *hard* filesystem isolation for "vibes-based" scoping that leaks on weak models. So: build the minimum scaffold that still *mechanically* holds the invariant.

**The one-liner:** *Invest in interfaces, not implementations; enforce with gates, not prose; delete scaffolding as models improve.*

Two supporting struts the skill should carry:

- **[sourced — Ronacher & Poncela]** *Route mechanical → agent, judgment → human.* Formatting, type errors, forbidden imports, missing tests → back to the agent via lints/CI. **Migrations, permission changes, dependency additions, reliability/SLO choices, and architecture/boundary changes → surface to a human.** *("The Friction Is Your Judgment," 2026 — talk verified; quotes need re-checking.)* Boundary changes are explicitly *judgment* calls — which is what protects the design rules from being silently eroded by N agents optimizing for "it runs." Their failure modes are reusable anti-patterns: *friction theater, late friction, prompt-only friction, stale friction, misrouted friction.*
- **[sourced — Karpathy / arithmetic]** *Why slop compounds and why prose isn't enough.* Independent per-step success multiplies: 10 steps at 90%/step ≈ 35% end-to-end; each added "nine" of reliability is its own project ("march of nines"). A markdown skill improves *averages, not guarantees* — so reliability-critical invariants need **hard gates** (deterministic verification), e.g. Stripe wrapping Claude Code to auto-run tests and reject failing branches; import-linter + ADR rules that make a whole failure class impossible ("prevent entirely, don't detect repeatedly"). *(Karpathy "march of nines" — attribute to a Karpathy primary, not the secondhand explainer; Stripe "minions" and Skills-Bench figures are secondhand — see §II.3.)*
