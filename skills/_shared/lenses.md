# Lenses — Parallel Dispatch

Four **pre-write** reviewer lenses (dispatched at workflow step 5), plus one **post-write auditor** dispatched at step 8.5 for `scaffold` after files are written. Each has a specific anchoring concern. The four pre-write lenses run **in parallel** for `assess`; optional but recommended for `harden` and `scaffold`. For `diagnose`, run only the lens that matches the hypothesized failure mode (run all four if hypothesis is unclear).

Hosts that lack a delegation primitive: fall back to running the lenses sequentially in the same head, switching role between passes. The discipline of *switching lens between passes* matters more than parallelism.

## Lens 1 — Cold-Context Agent

**Anchoring concern:** what a stateless, freshly-spawned LLM with no prior conversation actually sees and trips on.

**Prompt to apply:**
> You are a stateless coding agent invoked on this repository for the first time. You have read the loaded playbook(s) and the user's task. Identify everything that would cause you to: (a) burn tokens on exploration before producing output, (b) hallucinate a convention this repo never expressed, (c) pick the wrong file in a monorepo, (d) follow the middle of an instruction file and miss the periphery rules, (e) exceed your context budget. For each finding, name the artifact (file path, hook name, MCP method) and the severity 0–4.

**Failure modes this lens catches:**
- Ambiguous or missing build/test commands.
- Convention drift between AGENTS.md and code.
- AGENTS.md > 200 lines (W2).
- No nested AGENTS.md in monorepos (Codex 88-file pattern absent).
- Skill descriptions that don't trigger reliably (missing trigger words).
- Repo with no `docs/` tree or repo-map.
- Long files (>500 lines) that don't fit in one read.

## Lens 2 — Maintainer

**Anchoring concern:** is the agent surface itself maintainable, or drifting?

**Prompt to apply:**
> You are the human maintainer of this repo's agent surface. You inherited AGENTS.md, the skills/, and the hook config from someone who left. Identify everything that will rot, drift, or contradict in the next 90 days: autogenerated boilerplate, overlapping skill triggers, conflicting nested rules, deprecated APIs still listed, stale worked examples, undated reflection logs. Cite W1, W7, W8 from empirical-warnings.md where applicable. Severity 0–4 per finding.

**Failure modes this lens catches:**
- Autogenerated AGENTS.md (W1).
- Skill triggers that overlap or shadow each other.
- Stale `.cursorrules` next to current `.cursor/rules/*.mdc` (or equivalent legacy file).
- ADRs that contradict the current implementation.
- AGENTS.md and CLAUDE.md drift instead of one being a symlink to the other.
- `inspired_by` provenance pointing to broken or outdated sources.

## Lens 3 — Adversarial

**Anchoring concern:** prompt injection, exfiltration, tool abuse, sandbox escape.

**Prompt to apply:**
> You are an attacker who can submit a PR, file an issue, or write a comment that the agent will read. Identify every vector by which you could: (a) inject instructions into the agent via AGENTS.md / skills / hooks / issue body / file content the agent loads, (b) exfiltrate secrets via tool output or trace data, (c) abuse a tool to escalate privilege (e.g., a write-mode MCP method exposed where read-mode would suffice), (d) escape the sandbox (process sandbox on a dev laptop counts), (e) trigger runaway autonomy (agent edits production data, force-pushes main, drops a DB). Cite W5, W10 where applicable. Severity 0–4 with 4 reserved for active exploit paths.

**Failure modes this lens catches:**
- AGENTS.md is editable without review (W5).
- Secrets mounted inside the agent's writable workspace.
- MCP server exposes write methods (drop_table, force_push) without an approval tier.
- Process sandbox on developer laptops (W10).
- Hook config that doesn't reject `rm -rf`, force-push to main, or prod-DB writes.
- Untrusted MCP servers (third-party, not reviewed) wired in.
- No allow-list on network egress from sandbox.

## Lens 4 — Auditor

**Anchoring concern:** can a human reconstruct what the agent did and why?

**Prompt to apply:**
> You are a compliance reviewer or post-incident investigator. You need to reconstruct (a) which agent did what when, (b) what context it had loaded, (c) which approvals were taken/skipped, (d) what was the provenance of any released artifact, (e) what the rollback path is. Identify gaps in traces, append-only event histories, approval tiers, ownership of prompts/skills/policies, provenance attestations (SLSA), and incident-disclosure paths. Severity 0–4 with 4 reserved for unauditable changes that touched production.

**Failure modes this lens catches:**
- No trace metadata (model, prompt hash, tool-schema version, commit SHA).
- Append-only event history missing or unwritable.
- Free-text logs that can't be correlated to runs.
- No ownership listed for AGENTS.md / skills / hook config.
- Release artifacts ship without attestations.
- Approval tier is binary (allow / deny) rather than tiered (free / ask / forbidden).
- No incident-disclosure path documented in the repo.

## Post-write auditor (step 8.5, `scaffold` only)

**Anchoring concern:** did the writer actually apply the `harden` heuristics the chosen playbook(s) prescribe — or did some get silently skipped between confirmation and disk?

**Why a separate dispatch:** the four pre-write lenses ran at step 5, before anything was on disk. They can name what *should* be done; they cannot verify what *was* done. A self-attestation checklist in the scaffold output is the wrong fix — the writer rubber-stamps its own work. Dispatch a fresh-context sub-agent whose **only inputs** are (a) the playbook(s) for the chosen sub-surface(s), (b) the git diff of the scaffold write, and (c) the failing harden heuristics the writer claimed. The auditor has no access to the writer's reasoning, internal monologue, or pre-write lens findings — only the playbook and what landed on disk.

**Prompt to apply:**
> You are a fresh-context auditor. You did not perform this scaffold. You are given (1) the playbook(s) for the sub-surface(s) being scaffolded, (2) the git diff of the writes that just happened, (3) the user's stated harness inventory from step 4.5 (if applicable). Enumerate every `scaffold` heuristic **and** every `harden` heuristic in the loaded playbook(s) — scaffold heuristics govern what the scaffold itself should produce; harden heuristics fire whenever an artifact the scaffold created (e.g., AGENTS.md, a hook) is now in scope for them. For each, classify as one of: `applied` (cite the line/file in the diff that satisfies it), `skipped-because-X` (cite the explicit reason the writer or user gave; "not relevant" is not a reason — name *why* it isn't), or `deferred` (named in the scaffold output as follow-up). Anything that doesn't fit those three classifications is a **miss** — report it as such. Severity 0–4 per miss; severity 3+ surfaces to the user as a must-do before claiming the scaffold is complete.

**Failure modes this lens catches:**
- Prescribed symlink missing (`CLAUDE.md` → `AGENTS.md`).
- Harness-specific equivalents not produced for harnesses named in step 4.5 (e.g., Codex inventoried but no `--ask-for-approval` config produced).
- CI assertion mentioned in a harden heuristic but not wired in.
- Per-skill artifact required by the contract (e.g., `evals/run-static-checks.sh`) not written for a new skill the scaffold created.
- Static-check / hook configured but not registered in the harness's settings file (hook file exists but `.claude/settings.json` doesn't reference it).
- **Deny-list / pattern-matching hook scaffolded without a negative-case test fixture next to it** (gates.md `scaffold` H5). The hook file exists but no test file covers the variant matrix the hook is trying to block — the regression vector logged at `docs/reflection-log/2026-05-16-hook-regex-bypasses-round1.md`.
- **Hook predicate uses regex-on-string for non-trivial patterns** (gates.md `scaffold` H6). Refspec forms, `+`-refspecs, option terminators, long-form aliases, and wrapper prefixes silently bypass regex deny-lists; argv parsing makes the surface explicit.
- **Reflection log scaffolded but no `README.md` pointer to it, in a repo without AGENTS.md** (cross-cutting: applies whenever a reflection log was written). The log lands as an orphan: present on disk, invisible to any agent walking in fresh because nothing always-loaded references it.
- **In `rules-from-coding-agent-failures` scaffolds only:** AGENTS.md scaffolded without the Stage-0 substrate (no `docs/reflection-log/` directory + index, no README pointer) means the W1 ≥3-failures floor was skipped — the AGENTS.md cannot have been hand-curated from observed failures because the log doesn't exist yet. **This failure mode does NOT apply to `harden-repo-for-coding-agents` scaffolds** — that skill explicitly supports scaffolding AGENTS.md from project context with no reflection log; the auditor must check which skill ran the scaffold before flagging this. Also in EDAR contexts: a reflection-log README that conflates the recording bar with the promotion bar (≥3-entries language too close to "how to add an entry") causes reviewers to self-filter single observations as "not yet a pattern."
- **Artifact written without citing a template, or with `<placeholder>` markers remaining.** Every scaffolded file should map to a template under `templates/artifacts/<sub-surface>/` (see playbook §Templates section). The auditor verifies that (a) the scaffold-bundle's Proposed-files table cited a template path for each file or stated why no template applies, (b) no `<placeholder>` strings remain in the written file, and (c) the file's top-level shape (sections / required frontmatter fields / table columns) matches the template. Drift between template and artifact is a shape-compliance miss.

**Skip cases:** `assess`, `harden`, `diagnose` — no writes happened, so there is nothing to audit. Step 8.5 is `scaffold`-only.

## Preamble before dispatch

Before spawning sub-agents, emit a short user-facing preamble — 3–4 lines, no
more. Four-lens fan-outs (and the `assess + all` 10-agent inverted topology)
go silent for a while; the preamble converts that wait from a black box into
an anticipated reveal.

The preamble must name:

- **Lenses dispatched** (e.g., "cold-context-agent, maintainer, adversarial,
  auditor"; or the single lens chosen for `diagnose`).
- **Sub-surface(s)** under review, or `all` for the inverted fan-out.
- **Rough time estimate** — `~1–2 min` for a single sub-surface 4-lens pass;
  `~3–5 min` for the inverted 10-agent `assess + all` topology. Not a hard
  promise.
- **What to watch for in the output** — one sentence telegraphing the
  finding shapes the user should expect (typically driven by the dominant
  lens or the chosen intent).

Example (`assess`, single sub-surface):

```text
Dispatching 4 lenses (cold-context, maintainer, adversarial, auditor)
against the `instruction-surface` sub-surface. ~1–2 min.
Watch for: AGENTS.md > 200 lines (W2), autogenerated boilerplate (W9),
and injection vectors in always-loaded files (W5).
```

Example (`assess + all`):

```text
Dispatching 10 sub-surface agents (each running 4 lenses sequentially)
across the full sub-surface set. ~3–5 min.
Watch for: which layer (legibility / action / control) sets the ceiling
score, and any sub-surface that drags the layer down by ≥2 points.
```

Skip the preamble for `diagnose` with a single lens, or for hosts that
don't show streaming text. Don't substitute a long status spinner — the
value is the user knowing *what is being looked for*, not just that work
is happening.

## Dispatch template

```
Spawn N=4 sub-agents in parallel. Each agent receives:
  - The user's task (intent + sub-surface + repo context).
  - The loaded playbook(s) for the chosen sub-surface(s).
  - references/empirical-warnings.md.
  - references/core/severity-rubric.md.
  - Its specific lens prompt from this file.

Each agent produces:
  - A list of findings, each with {severity, sub-surface, artifact-pointer, one-line description, lens-name}.
  - Optional: hypothesis ranking (diagnose only) or recommendation (harden / scaffold).

The main agent then synthesizes:
  1. Deduplicate findings that ≥2 lenses raised; bump severity to the highest reported.
  2. Preserve lens-specific findings (no merge).
  3. Surface conflicts (lens A says do X, lens B says don't) as open questions.
  4. Emit the intent-specific template.
```

### Post-write dispatch (step 8.5, `scaffold` only)

```
Spawn 1 sub-agent in a fresh context. The agent receives:
  - The playbook(s) for the chosen sub-surface(s) (full text).
  - The git diff of the writes that just happened.
  - The harness inventory collected at step 4.5 (if applicable).
  - The "Post-write auditor" prompt from this file.

It does NOT receive:
  - The pre-write lens findings.
  - The writer's reasoning or scaffold-bundle output.
  - The conversation history.

The agent produces:
  - For each `harden` heuristic in the loaded playbook(s):
    {heuristic-id, classification: applied | skipped-because-X | deferred | miss,
     evidence: file:line for `applied` / explicit reason for `skipped` / follow-up
     pointer for `deferred` / "no evidence in diff" for `miss`, severity 0–4}.

The main agent then:
  1. Lists every `miss` to the user with severity, before claiming the scaffold is complete.
  2. Asks the user to either accept the misses as `skipped-because-X` (with reason),
     accept as `deferred` (with follow-up pointer), or amend the scaffold.
  3. Does not silently downgrade misses; either the user resolves them or they
     stay flagged in the scaffold-bundle output.
```

## Multi-surface fan-out

For `assess + all` (assessing the whole repo, not one sub-surface), invert the topology: dispatch **one sub-agent per sub-surface** (10 agents), each running the 4 lenses sequentially inside itself. The main agent then synthesizes across sub-surfaces using the maturity rubric to score each layer.

## Skip cases

- Tiny copy edits to a single playbook → no dispatch.
- Deterministic command verification → no dispatch.
- Tasks requiring secrets or live production access → no dispatch (sub-agents inherit no privileged context).
