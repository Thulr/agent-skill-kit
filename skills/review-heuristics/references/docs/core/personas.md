# Audience Personas

Use these as lenses, not stereotypes. A single docs surface often serves several at once.

## Primary personas

- **First-time developer integrator** — wants first success fast, with install, credentials, one runnable example, and a clear next step.
- **Working developer maintainer** — already shipped something; needs reference accuracy, changelogs, migration notes, troubleshooting, version specificity, and search.
- **End user in recovery** — is blocked inside a product; needs plain language, visible next action, preserved work, and minimal context switching.
- **Exploratory end user** — is learning a feature; needs progressive disclosure, empty-state teaching, and help in the moment of need.
- **Coding agent** — retrieves chunks, reads schemas, follows instructions literally, retries quickly, and cannot infer hover/visual-only context.
- **Runtime agent or workflow** — calls APIs/tools autonomously; needs stable codes, idempotency, rate limits, scopes, costs, side effects, and rollback semantics.
- **Support/content operator** — owns the feedback loop; needs article analytics, zero-result searches, support-ticket joins, stale-page detection, and docs backlog hygiene.
- **Platform/product owner** — decides what must block release; needs risk ranking, ownership, verification method, and measurable target.

## Persona use

1. Name the primary persona before scoring.
2. Name secondary and harmed personas when the same change affects multiple audiences.
3. Do not average conflicting needs. Route conflicts through `references/docs/playbooks/audience-conflicts.md`.
4. If evidence is missing, say which persona needs validation and what signal would confirm it.
