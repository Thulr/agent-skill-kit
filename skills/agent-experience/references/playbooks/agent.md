# Agent Playbook

## Scope

AI-agent and coding-assistant readiness: `AGENTS.md` (and harness-specific
mirrors), MCP servers, retrieval-friendly documentation structure,
snippet-grade examples, and the failure modes coding agents hit on
unfamiliar repos. Distinct from human-facing developer reference and
editor-integration surfaces: this playbook covers the surface developers
consume *through* an agent — Claude Code, Cursor, Codex, Copilot, Windsurf,
Aider — when that agent is the integrator's daily driver. For the human
developer-experience surfaces this sits alongside (content structure, editor
integration, snippet samples, contributor onboarding), see
`dx-critique` / `dx-design` (`docs`, `ide`, `examples`, `contributor`).

## Grounding

- **AGENTS.md convention (agents.md)** — a single hand-curated repo-root
  file that coding agents read first to learn project context, commands,
  invariants, and load-bearing rules. Hand-curated beats autogenerated:
  the file is a contract with future agents, not a snapshot of current
  code.
- **Anthropic — Model Context Protocol (MCP) specification** — a uniform
  protocol for exposing tools, resources, and prompts to agents. Speaking
  MCP lets every conforming agent (Claude Code, Cursor, future hosts)
  access the same capabilities without per-host plugins.
- **Mündler et al. — *Evidence-Driven Agent Scaffolding* (LogicStar / ETH,
  Feb 2026)** — scaffolding written from fewer than three observed agent
  failures produces plausible boilerplate that hurts agent success rates;
  rules must be promoted from a logged pattern of failures, not invented.

## Good signals

- An `AGENTS.md` exists at the repo root, hand-curated, and names the
  project, layout, commands, invariants, and load-bearing rules.
- Harness-specific entry points (`CLAUDE.md`, `.github/copilot-instructions.md`,
  `.cursor/rules`) either exist independently or symlink to `AGENTS.md` so
  every supported agent reads the same canonical content.
- The file is hand-curated; an explicit "do not autogenerate" instruction
  exists where the team has decided autogeneration produces noise.
- Where the product offers an MCP server, the server is published, documented,
  and exercised in CI; agents can connect without bespoke setup.
- Documentation pages are structured so retrieval works: meaningful headings,
  short focused paragraphs, code blocks tagged with the language, and
  snippet-grade examples that stand alone.
- The product's examples are paste-runnable in a fresh agent session — no
  hidden state, no implied prior commands, no out-of-band setup.
- Agent-relevant hooks, gates, or sandboxing rules are documented so an
  agent does not trip on a destructive-command block it does not know about.
- A reflection log or equivalent captures observed agent failures; rules
  promoted to `AGENTS.md` trace back to at least three logged incidents.
- The repo declares whether the project is agent-friendly (welcomes
  agent contributions) or agent-hostile (humans only, gates enforced); the
  policy is explicit, not implicit.

## Common failures

- No `AGENTS.md`; agents fall back to scanning the README and guessing at
  project conventions, often wrong.
- `AGENTS.md` is autogenerated (`/init`, `/Generate Cursor Rules`,
  similar) without curation; agents read confidently wrong instructions
  and act on them.
- Multiple harness-specific files (`CLAUDE.md`, `copilot-instructions.md`,
  `.cursor/rules`) exist but drift apart; different agents learn different
  things about the same repo.
- The MCP server exists but is undocumented, not exercised in CI, or
  requires bespoke per-host setup that defeats the protocol's value.
- Documentation is one wall of prose per page; retrieval surfaces irrelevant
  context or misses the key snippet.
- Examples assume prior session state ("after you've created the client
  above"); an agent invoked cold cannot reproduce the example.
- Hooks or sandboxing rules block commands the agent will plausibly try
  (`git push --force`, `rm -rf`), but no `AGENTS.md` entry warns of the
  block; the agent retries the same command and reports stuck.
- Rules in `AGENTS.md` are invented from one observed failure or from a
  generic best-practice list; they constrain behavior without an evidence
  base and accumulate noise.
- No statement on whether agent-authored contributions are welcome; agent
  PRs sit in review limbo or get rejected with no traceable policy.

## Heuristics

- **Hand-curated `AGENTS.md`** *(design, audit)* — `AGENTS.md` is
  hand-written and contains project context, commands, invariants, and
  load-bearing rules; an explicit "do not autogenerate" instruction
  forestalls drift. Symlinks for harness-specific mirrors keep one source
  of truth.
- **Harness-mirror parity** *(audit)* — `CLAUDE.md`,
  `.github/copilot-instructions.md`, and `.cursor/rules` either symlink to
  `AGENTS.md` or are tested in CI to match it. A check fails the build if
  they diverge.
- **Evidence-promoted rules** *(design, audit)* — load-bearing rules in
  `AGENTS.md` trace back to a reflection-log entry or equivalent record
  with three or more observed failures. Rules invented from a single
  incident or from generic best practices are dropped.
- **MCP where it earns its keep** *(design, audit)* — if the product
  exposes tools an agent would invoke, those tools are published as an
  MCP server, documented in `AGENTS.md`, and exercised in CI. If MCP does
  not fit, that decision is documented so agents do not look for what is
  not there.
- **Retrieval-friendly docs** *(design, audit)* — documentation pages
  have meaningful headings, focused paragraphs, language-tagged code
  blocks, and snippet-grade examples that stand alone. Retrieval surfaces
  the right chunk; agents reason from it without further context.
- **Cold-start runnable examples** *(audit, design)* — every example runs
  in a fresh agent session with no implied prior state. "After you've set
  up the client" is a gap; the example should set up the client itself.
- **Documented hooks and gates** *(audit, design)* — destructive-command
  blocks, sandboxing rules, and CI gates that the agent will encounter are
  named in `AGENTS.md` with the rationale. Surprise gates produce
  retry-loop failures that look like the agent is stuck.
- **Explicit agent-contribution policy** *(design)* — the repo states
  whether agent-authored PRs are welcome, what gates they must pass, and
  who reviews them. Implicit policies create review-loop friction.

## Quick diagnostic

| Question | If no | Action |
| --- | --- | --- |
| Is there a hand-curated `AGENTS.md` at the repo root? | Agents guess at conventions | Author `AGENTS.md`; do not autogenerate |
| Do harness mirrors stay in sync? | Different agents learn different things | Symlink or CI-check parity |
| Are load-bearing rules tied to observed failures? | Rules drift toward noise | Adopt a reflection-log workflow; promote rules from ≥3 entries |
| Is the MCP server (if any) published, documented, and CI-exercised? | Agents cannot use it reliably | Publish, document, and exercise in CI |
| Are docs retrieval-friendly? | Wrong chunks surface | Restructure with meaningful headings and focused paragraphs |
| Do examples run cold in a fresh agent session? | Hidden state breaks reuse | Make every example self-contained |
| Are hooks and gates documented? | Agents loop on blocked commands | List gates in `AGENTS.md` with rationale |
| Is the agent-contribution policy explicit? | Review limbo for agent PRs | State the policy in `CONTRIBUTING.md` and `AGENTS.md` |
