# Spec — split the failure-driven workflow out of `project-agentification`

**Date:** 2026-05-16
**Status:** draft (awaiting reviewer approval before execution)
**Triggered by:** maintainer review of W1's evidence chain (see Background)

## Problem

`project-agentification` currently treats `AGENTS.md` as a failure-derived
artifact. Concretely, the skill:

- Refuses to scaffold any instruction file (`AGENTS.md`, rules, hooks)
  without **≥3 observed agent failures stated** (W1 floor).
- Imposes a hard bootstrap order: **Stage 0** (reflection log + README
  pointer) → **Stage 1** (`AGENTS.md`) → **Stage 2+** (everything else).
- Requires every `AGENTS.md` rule to **cite a reflection-log entry** as its
  evidence source.
- Cites Mündler et al. (arXiv:2602.11988) "~3% success drop, >20% cost
  inflation" as the load-bearing justification.

This framing is borrowed from harness-engineering literature (Engineering
Agents — Harness Assessment; the Mündler benchmark), where success rate and
token cost are measurable signals because the audience runs eval suites and
benchmarks. **For the actual audience of this skill — repos that want to be
agent-friendly — the feedback signal does not exist.** Most projects have
no benchmark, no A/B baseline, no run-level telemetry; the only signal is
the qualitative "the agent did the wrong thing" observation.

Consequences:

1. **The ≥3 floor is unenforceable in spirit.** A repo that sees 5 agent
   sessions a month could wait months before three same-class failures
   accumulate. Meanwhile no `AGENTS.md` gets written, even though one
   would help.
2. **The Mündler citations are faith-based at the application layer.** A
   downstream user cannot reproduce the measurement; they take the rule on
   authority.
3. **`AGENTS.md` is being mis-cast.** For non-harness projects it is a
   conventions doc (stack, layout, commands, monorepo scope) — exactly the
   pattern of the `vercel/next.js` `AGENTS.md` this skill cites as
   canonical, which has nothing to do with a failure log.
4. **Manual reflection-log curation is friction at the moment the skill
   should be removing friction.** The skill's whole premise is "make a
   repo agents can be turned loose on"; gating that on a separate
   incident-log workflow contradicts the premise.

The maintainer's critique: *"AGENTS.md generally contain project context —
they're not reserved or generated by failures — at least not for non-harness
projects."* That critique lands.

## Constraints

- **Don't lose the work landed in PR #11.** The per-file reflection-log
  shape is correct for *this repo* (a skill catalog where skill efficacy
  IS the thing under measurement). It needs to keep working; the open
  question is whether the *workflow it implements* belongs in
  `project-agentification` or somewhere else.
- **Two audiences must be served:** (a) any repo wanting to be agent-ready
  — needs project-context-first scaffolding without failure-log
  prerequisites; (b) teams operating/building agent infrastructure —
  legitimately need the evidence-driven workflow because the signal exists.
- **Cross-skill primitives shouldn't be duplicated.** The four lenses
  (cold-context, maintainer, adversarial, auditor), parts of the
  empirical-warnings, and potentially parts of the maturity rubric apply
  to both audiences. Copy-paste between two skills creates drift; the
  earlier inline-validators-duplicated-four-ways failure
  (`docs/reflection-log/2026-05-15-trigger-evals-schema-drift.md`) is the
  cautionary tale.
- **Honor the existing repo conventions.** Skills are self-contained
  installable units under `skills/<name>/`; this constraint shapes how
  shared content can be packaged.

## Approach

**Split into two skills + a shared cross-skill primitives directory.**

### `project-agentification` (reframed)

Audience: any repo that wants coding agents to work well in it. Assumes
no eval infrastructure, no benchmark, no measured baseline.

Owns:
- Project-context-first `AGENTS.md` scaffolding (WHAT / WHY / HOW)
- Gates sub-surface (hooks, CI, sandbox modes)
- Skills, tools, telemetry, docs-index, instruction-surface sub-surfaces
- Harness-inventory step (4.5) and the per-harness equivalents
- Skills-vs-`AGENTS.md` tradeoff guidance

W1 reframes from "≥3 observed failures required" to "don't `/init` — write
project-specific content from project knowledge." The rationale shifts from
*measured success delta* to *content quality verifiable by reading the
file*. The ≥3 floor is removed entirely.

### `agent-rules` (new skill)

Audience: teams building or operating coding-agent infrastructure where
eval suites, run-level telemetry, or A/B baselines exist (agent product
teams, harness developers, skill catalogs under test). Skill catalogs
like this very repo are an instance of the audience.

Owns:
- Reflection-log workflow (directory shape, recording-bar / promotion-bar,
  `_template.md`, README index)
- W1 ≥3 promotion floor + Mündler citations
- Bootstrap order (Stage 0 / 1 / 2)
- Evidence-driven scaffolding heuristics
- Maturity rubric Level 4+ portions (Spec Architecture, Sovereign
  Engineering) that assume harness infrastructure

### Shared cross-skill primitives directory

The four lenses, parts of `empirical-warnings.md` that apply to both
audiences (e.g., W2 length, W3 hooks-over-prose, W5 attack surface, W6
token budget, W10 sandbox), and potentially the cross-cutting parts of the
maturity rubric (Level 1–3 legibility/action signals) live in a shared
location both skills reference.

**Proposed location:** `skills/_shared/` (leading underscore signals "not
a standalone skill"; analogous to the existing `.experimental/` lane but
non-installable). Both skills' `references/` files include relative-path
references to `../../_shared/<file>.md`.

The shared directory is **the spec's biggest open question** — see Open
Questions below.

## Acceptance criteria

A reviewer should be able to verify the following after the migration PR(s)
land:

1. `project-agentification`'s SKILL.md has no Stage 0/1/2 bootstrap order;
   no "scaffold refuses without ≥3 failures stated" workflow step; no
   reflection-log sub-surface entry in `layer-router.csv`.
2. `project-agentification`'s AGENTS.md template has no "every rule must
   cite a reflection-log entry" requirement; the `### Rule N` template
   does not require a log-entry citation in the heading.
3. The Mündler "~3% / >20%" claim does not appear in
   `project-agentification` as load-bearing evidence. (It may appear as
   "see also `agent-rules`" cross-reference.)
4. A new skill exists at `skills/.experimental/agent-rules/`
   with a valid `SKILL.md`, `skill.json`, `evals/`, that passes
   `just check`.
5. The four lenses live in exactly one canonical location at maintenance
   time (`skills/_shared/lenses.md`), referenced by both consuming skills
   via relative symlinks. At install time, `npx skills` dereferences the
   symlinks and ships each skill self-contained with a regular-file copy
   of `lenses.md` — verified by scratch install. The single-source
   invariant holds at maintenance time, not at install time; downstream
   users get the canonical content but not the directory structure.
6. `docs/reflection-log/` in this repo continues to function (paths
   resolve, README is accurate). Cross-references in this repo's
   `AGENTS.md` and `README.md` point to `agent-rules` for
   the workflow doc, not `project-agentification`.
7. The new skill scaffolds the reflection log into target repos exactly as
   `project-agentification` does today (no behavioral regression for the
   reflection-log-using audience).
8. `npx skills add . --list` shows both skills.
9. Downstream install of `project-agentification` alone does **not** pull
   in the reflection-log artifact templates (or `_shared/` content that
   only `agent-rules` consumes).

## Open questions

All resolved 2026-05-16. Plan execution is unblocked.

### Q1. Shared directory mechanics — RESOLVED

**Decision:** **symlink-as-canonical.** Canonical source lives at
`skills/_shared/<file>.md`. Each consuming skill has
`<skill>/references/<file>.md` as a **relative symlink** to the canonical
source. At install time, `npx skills` dereferences the symlink and ships
the resolved file content to the downstream user — they get a regular
file (no symlink, no missing reference). At maintenance time, editing
the canonical file propagates to all consumers automatically. No lint
script needed; drift is impossible because the symlink IS the same
inode.

**Why this is the chosen mechanism** (verified empirically — installed
both a normal skill and a skill containing a cross-directory symlink
into scratch dirs to observe `npx skills@1.5.7` behavior):

- Test 1: installed `project-agentification` alone. Confirmed the
  installer only discovers SKILL.md-bearing dirs (`Found 7 skills`) and
  copies each skill's own tree to `./.agents/skills/<name>/`. A sibling
  `skills/_shared/` with no SKILL.md is silently ignored as a skill.
- Test 2: created a tiny test repo with `skills/test-skill/references/canonical.md`
  as a symlink to `../../_shared/canonical.md`. Installed via
  `npx skills add`. The installed file at
  `./.agents/skills/test-skill/references/canonical.md` was a **regular
  file** (not a symlink, not broken) containing the canonical content.
  → The installer dereferences symlinks during extraction.

So at the downstream layer, the user sees a self-contained skill with
regular files — the `_shared/` directory is purely a maintenance-side
single-source-of-truth that disappears at install. Best of both worlds.

**Options ruled out:**

- **Per-skill embedded copy + lint.** Strictly worse: needs a lint
  script, allows drift between lint runs, requires maintainer discipline
  to update both files. The symlink approach gets the same install
  behavior with none of that overhead.
- **Defer the shared dir.** Ships broken cross-refs ("see
  `project-agentification` §Lenses" from a separately-installed
  `agent-rules`).
- **Copy-at-install via `npx skills` tool changes.** This repo doesn't
  own the tool.
- **Owning our own installer** (like BMAD-METHOD's `npx bmad-method`).
  The cost (months of work to reimplement the 55+ harness matrix the
  `skills` tool already handles) is upside-down vs. the benefit of
  sharing 2–3 primitives. Stay on `npx skills`.

**Scope discipline for `_shared/`:** only put content there that is
**genuinely identical** across consumers. If the two skills end up
needing slightly different copies of a primitive, the canonical-source
pattern stops fitting and divergence becomes silent — drop it from
`_shared/` and maintain independently. Q2/Q3 above apply this filter.

**Relative-symlink paths to canonical** (computed per consuming skill):

- `skills/<published>/references/<file>.md` →
  `../../_shared/<file>.md` (2 levels up)
- `skills/.experimental/<name>/references/<file>.md` →
  `../../../_shared/<file>.md` (3 levels up)
- `.agents/skills/<name>/references/<file>.md` →
  `../../../skills/_shared/<file>.md` (3 levels up + across)

### Q2. Maturity rubric scope — RESOLVED

**Decision:** **split — Levels 1–3 in `project-agentification`; Levels
4–5 in `agent-rules`.** The rubric lives in each skill's
own `references/core/maturity-rubric.md`; it does **not** go in
`_shared/` because each skill ships a different subset (Q1 scope
discipline: only put genuinely-identical content in `_shared/`).

Cross-link in both directions: `project-agentification`'s rubric ends
with "Levels 4–5 (Spec Architecture, Sovereign Engineering) require
infrastructure outside this skill's scope — see
`agent-rules`." `agent-rules`'s rubric
opens with "Levels 1–3 are described in `project-agentification`'s
maturity rubric; this file extends with Levels 4–5."

Rationale: Levels 1–3 are about whether `AGENTS.md` exists and whether
hooks enforce constraints — any repo can score against them. Levels 4–5
assume specs, eval pipelines, cost tracking, model routing, SLSA — all
harness-engineering concerns where the feedback signal exists.

### Q3. `empirical-warnings.md` split — RESOLVED

**Decision:** **W2–W10 to `_shared/empirical-warnings.md`; W1 stays
sole-tenant in `agent-rules/references/empirical-warnings-w1.md`.**

W2/W3/W4/W5/W6/W7/W8/W9/W10 apply identically to both audiences
(content quality, length budgets, hooks-over-prose, attack surface,
sandbox, etc.) — these meet the "genuinely identical" filter for
`_shared/`. W1 (the ≥3 floor + Mündler citations) is the failure-driven
core of `agent-rules` and does **not** fit either skill
the same way.

**Numbering:** keep W1 as a per-skill standalone file (`empirical-warnings-w1.md`)
in `agent-rules` so the shared file numbers cleanly W2–W10
without gaps. Both skills' `SKILL.md` and playbooks reference warnings by
their canonical W-number (W1, W2, …); the file split is invisible to
citations.

Cross-link: `agent-rules/references/empirical-warnings-w1.md`
ends with "W2–W10 live in `_shared/empirical-warnings.md` — same numbering."

### Q4. Reflection-log artifact templates — RESOLVED

**Decision:** **move `templates/artifacts/reflection-log/` to
`agent-rules`; `project-agentification` keeps a one-line
"see also" pointer in its `governance` playbook** (where reflection log
already gets a brief mention).

Discoverability rationale: a user `assess`-ing a repo via
`project-agentification` who finds they have a feedback signal worth
capturing should be told where to go next. Silence makes the
evidence-driven workflow undiscoverable from inside the
`project-agentification` audience.

The pointer is one line, not a playbook section — anything heavier
re-imports the audience confusion this spec is trying to remove.

### Q5. Lens worked-example citations — RESOLVED

**Decision:** **genericize the worked-example citations in
`_shared/lenses.md`** so the shared copy makes sense for any consumer.

Concretely: replace "the regression vector logged at
`docs/reflection-log/2026-05-16-hook-regex-bypasses-round1.md`" with "the
regex→argv hardening bypass family (documented in any repo tracking it)."
The canonical example stays self-contained — readers don't need access
to this repo's specific reflection-log files to understand what the lens
catches.

No annotated this-repo-only copy is needed; the genericized text is
clear enough.

### Q6. Naming sanity check on `agent-rules` — RESOLVED

**Decision:** **keep `agent-rules`.** Maintainer
selected it during initial sequencing discussion; long but
self-descriptive. No alternative was a clear improvement.

## Background

This spec was motivated by maintainer review of W1's evidence chain on
2026-05-16, immediately after PR #11 landed the per-file reflection-log
restructure. The relevant exchange:

> Maintainer: "Who tracks the success % and token cost? I'm wondering if
> this is conflating someone's project, that they're going to use an
> agent harness to do the development, and the agent harness itself...
> If you're working on a harness repo itself I could see tracking agent
> failures because you'd be tracking tokens and % failures."

> Maintainer: "AGENTS.md generally contain project context — they're not
> reserved or generated by failures — at least not for non-harness
> projects."

The skill author (assistant) agreed the layer mismatch was real: research
evidence (Mündler benchmark) was being cited at the application layer
(individual repos), where the feedback signal that justifies the rule
doesn't exist.

See also:
- `docs/reflection-log/2026-05-16-recording-bar-conflated-with-promotion-bar.md`
  — the earlier reflection-log-shape critique that this spec extends.
- PR #11 — merged 2026-05-16 — restructured `docs/agent-failures.md` →
  `docs/reflection-log/`. Did not address the layer mismatch; this spec
  does.
