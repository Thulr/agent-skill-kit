# Evidence-Driven Agent Rules

Capture observed agent failures, promote the recurring ones into rules, and score
advanced (Level 4–5) agent-readiness. For repos with a *feedback signal*: eval suites,
run-level telemetry, A/B baselines, or a skill catalog under test. No signal yet — no
benchmark, no telemetry baseline, no "we watched the agent do X" stream? Use
`harden-repo-for-coding-agents` alone; promotion here needs evidence to drive it.

## Boundaries

Do NOT use for first-pass agent-readiness scaffolding when no feedback signal
exists yet (use harden-repo-for-coding-agents), or to design or operate an AI
product's eval and optimization loops (use agent-test / agent-ops).

**Produces:** `capture` writes the reflection-log scaffold (`docs/reflection-log/README.md`,
`_template.md`, and a repo-root `README.md §Agents` pointer); `triage` clusters existing
entries into a failure-mode ontology and closure queue; `promote` proposes the smallest
rule / hook / CI-gate / skill / eval diff plus the entries it cites; `closeout` verifies
implemented closure work and marks entries resolved; `assess-l4l5` scores Level 4–5
maturity with a ceiling and prioritized gaps. Tracked `assess-l4l5` runs also emit
`rules-from-coding-agent-failures-findings-ledger-<date>-<slug>.md` and
`rules-from-coding-agent-failures-workflow-state-<date>-<slug>.json`.

## Core principle

**Single observations are cheap to record; rules cost trust to ship.** Recording is
low-friction and always worth it once you can write a non-trivial *What to do
differently* line. Promotion — turning logged entries into an `AGENTS.md` rule, hook, or
CI gate — is gated on **three or more entries describing the same gap**, because
scaffolding from one observation overfits to plausible boilerplate (W1, Mündler et al.,
arXiv:2602.11988, Feb 2026: autogenerated context files drop task success ~3% and inflate
cost >20%). The recording bar and the promotion bar are deliberately different; the
earlier single-file log conflated them and agents self-filtered entries worth keeping.
When in doubt, record. `triage` searches later.

**A log is not a loop until it changes the next run.** A pile of reflection entries is
telemetry, not improvement, until the entries become a durable closure surface: a rule,
skill patch, eval, test, hook, CI gate, source-doc fix, or roadmap package with a
verification rule.

## Quickstart

Begin with `capture` to scaffold the log:

> `Run capture: scaffold the reflection log for this repo.`

It loads `references/playbooks/reflection-log.md`, writes `docs/reflection-log/README.md`
and `_template.md`, and adds a repo-root `README.md §Agents` pointer so the log is
discoverable from an always-loaded surface. **Precondition:** if `AGENTS.md` exists but
does not link the log, `capture` refuses. Add one line under its references section and
re-run:

```markdown
- [docs/reflection-log/](./docs/reflection-log/) — per-failure entries; rules in this file may cite them.
```

`triage`, `promote`, `closeout`, and `assess-l4l5` come later: they assume Stage 0 (this log) and Stage 1
(AGENTS.md from `harden-repo-for-coding-agents`) are already in place. Full staging in
[`references/bootstrap-order.md`](./references/bootstrap-order.md).

## Activation

- **Bare invocation** (`"set up reflection log"`, `"use rules-from-coding-agent-failures"`):
  show a compact menu: mode choice (guided / autopilot / grill me?) and numbered
  intents from the router. Wait. No file inspection, no network calls, no writes.
- **Concrete invocation** with intent inferable: skip to Workflow step 2.
- **Ambiguous invocation**: ask one — e.g., *"Are you capturing a new failure entry,
  triaging a pile of entries, promoting a pattern into a durable change, closing out
  resolved entries, or scoring Level 4-5 maturity?"* or
  *"Is this for a single incident post-mortem or a recurring feedback loop?"*

## Workflow

1. **Pick intent.** `capture` (scaffold the log + README pointer), `triage` (turn many
   entries into failure-mode clusters and work packages), `promote` (turn a cluster into
   the smallest durable change), `closeout` (verify closure work and mark entries
   resolved), or `assess-l4l5` (score Level 4–5 maturity). Ambiguous → ask once.
2. **Load context.** Always: `references/empirical-warnings-w1.md` and
   `references/playbooks/reflection-log.md`. For `triage`, `promote`, or `closeout`,
   also load `references/trackable-findings.md` and read the whole log
   (`docs/reflection-log/[0-9]*.md`). If the closing change is a hook, CI check, static
   validator, or branch-protection rule, also load `references/playbooks/gate-hardening.md`.
   For `assess-l4l5`, also load `references/core/maturity-rubric.md`.
3. **capture.** Scaffold `docs/reflection-log/README.md` and `_template.md` from
   `templates/artifacts/reflection-log/`; add the `README.md §Agents` pointer if absent;
   refuse if `AGENTS.md` exists without a link to the log.
4. **triage.** Cluster entries first by `sub-surface:`, then by root cause / missing
   context / missing verification / unsafe action / skill failure / tool limitation.
   Emit `templates/artifacts/reflection-log/triage-report.md`: clusters, entry filenames,
   likely closure surface, priority, and one verification rule per cluster. Do not write
   AGENTS.md, hooks, skills, or issues during triage unless the user explicitly asks.
5. **promote.** For a chosen cluster with three or more entries describing the same gap,
   propose the smallest durable change: AGENTS.md rule, `SKILL.md` patch, playbook update,
   eval/trigger case, test fixture, hook, CI gate, docs/source fix, or roadmap package.
   Confirm before writing unless the user requested autopilot. Refuse below the W1 floor
   unless the output is only a draft closure plan. For a gate, require the
   **gate-hardening variant matrix** and a regression fixture before calling it done.
6. **closeout.** Given a closure plan, PR, diff, ledger, or list of entries, re-run each
   verification rule. Mark entries `resolved` only when evidence exists; otherwise leave
   them open and label the cluster `blocked` or `needs_evidence`. Emit
   `templates/artifacts/reflection-log/closeout-report.md` plus a since-last-run
   scoreboard from `references/trackable-findings.md` when prior state exists.
7. **assess-l4l5.** Assume Levels 1–3 are already scored by `harden-repo-for-coding-agents` (confirm
   with the user); score Levels 4–5 against the rubric; report ceiling and gaps with
   stable IDs (`ED-L4L5-NNN`) per `references/trackable-findings.md`.
8. **Emit, citing the entries you drew on** (filenames plus their *What to do
   differently* lines):
   - `capture` → files written, the §Agents pointer diff, a validation checklist.
   - `triage` → failure-mode clusters, priority, closure surface, verification rule, and
     next safe action for each cluster.
   - `promote` → the proposed rule/hook/skill/eval/test/docs/gate change, the three-plus
     entries that justify it, and the diff for confirmation.
   - `closeout` → verification evidence, entries moved to resolved, entries left open,
     blockers, and since-last-run scoreboard when available.
   - `assess-l4l5` → maturity score per layer, ceiling, prioritized gaps.
9. **Create tracking state.** For `assess-l4l5` with 7+ gaps, a level-ceiling blocker, or
   a save/track request, write both: a ledger at
   `docs/audits/rules-from-coding-agent-failures-findings-ledger-<YYYY-MM-DD>-<scope-slug>.md` and workflow
   state at `docs/audits/rules-from-coding-agent-failures-workflow-state-<YYYY-MM-DD>-<scope-slug>.json` (fall
   back to `audit-artifacts/rules-from-coding-agent-failures-{findings-ledger|workflow-state}-<YYYY-MM-DD>-<scope-slug>.{md|json}`
   if `docs/audits/` is unwritable). Report both paths. Roadmaps, issues, and promotion
   changes still need confirmation.

> **Wrong direction?** If the user says this isn't what they meant, go back to
> Understand (step 1) — do not patch in the wrong direction. Restate the
> corrected understanding and re-plan.

## Modes

Guided Draft (default), Autopilot, Grill Me — contract in
[`references/modes.md`](./references/modes.md). For `promote`, default to **Grill Me**:
shipping a rule is expensive trust-wise, so open the trade-offs rather than infer them.

## Output requirements

Every output cites the reflection-log entries it draws on — filenames plus their
*What to do differently* lines — names the intent, and carries that intent's load-bearing
section (Workflow step 8). For `triage`, every cluster needs a closure surface and
verification rule; for `promote`, include the three-entry floor and, for a gate, the
gate-hardening variant matrix and a regression fixture; for `closeout`, include evidence
before changing statuses; for `assess-l4l5`, stable `ED-L4L5-NNN` IDs.
